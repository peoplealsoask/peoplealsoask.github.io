Nginx 502 Bad Gateway Error: What It Is & 7 Proven Fixes (2026 Guide)

Learn what the Nginx 502 Bad Gateway error means, its common causes, and step-by-step fixes. Includes server, proxy, and PHP-FPM troubleshooting.

Executive Summary: The Nginx 502 Bad Gateway error indicates that Nginx received an invalid response from an upstream server. Fix it by checking backend services, proxy settings, PHP-FPM, and server resources.

The Nginx 502 Bad Gateway error occurs when Nginx, acting as a reverse proxy, receives an invalid or no response from an upstream server (e.g., PHP-FPM, Apache, Node.js, or a load balancer). This is a server-side error, meaning the problem lies with the backend service, not your browser. Common causes include a crashed PHP-FPM process, exhausted server resources, misconfigured proxy timeouts, or a faulty upstream server. To resolve it, start by restarting your backend services, checking error logs, and ensuring your server has enough memory and CPU. Below, we dive into seven proven fixes.

# What Is the Nginx 502 Bad Gateway Error?

The 502 error is an HTTP status code meaning “Bad Gateway.” In Nginx, this happens when the server is set up as a gateway or proxy to another server (the upstream). If that upstream server is down, overloaded, or returns a malformed response, Nginx serves a 502. Unlike a 504 (Gateway Timeout), which indicates the upstream took too long to respond, a 502 usually points to a connectivity or processing failure.

# Common Causes of the 502 Error

  • PHP-FPM Crash: The most frequent culprit, especially on dynamic sites using PHP.
  • Upstream Service Down: The backend web server (Apache, Node, Gunicorn) is not running.
  • Resource Exhaustion: Running out of memory or file descriptors.
  • Proxy Buffer Issues: Buffer sizes misconfigured for large responses.
  • Firewall Blocking: A firewall on the upstream server is blocking Nginx’s IP.
  • DNS Resolution Failure: Nginx cannot resolve the upstream hostname.

# 7 Proven Fixes for the Nginx 502 Bad Gateway Error

# 1. Restart PHP-FPM and Nginx

Often a simple restart clears transient issues. On systemd systems, run:

sudo systemctl restart php-fpm
sudo systemctl restart nginx

After restarting, test the site. If it loads, the issue was likely a temporary PHP process hang.

# 2. Check PHP-FPM Status and Logs

Verify PHP-FPM is running: sudo systemctl status php-fpm. Look for “Active: active (running)”. If it’s inactive or failed, dive into the logs (/var/log/php-fpm/error.log). Common errors include memory limit exhaustion or a crashed pool. Adjust pool settings in /etc/php-fpm.d/www.conf, increasing pm.max_children if you see “server reached pm.max_children setting.”

# 3. Review Nginx Error Logs

Nginx logs errors to /var/log/nginx/error.log. Use tail -f to watch real-time. You might see “no live upstreams” (upstream servers are all down) or “upstream prematurely closed connection” (PHP-FPM crashed).

# 4. Validate Upstream Server Configuration

In your Nginx site config (e.g., /etc/nginx/sites-available/example.com), ensure the proxy_pass directive points to a running service. For example:

location / {
    proxy_pass http://localhost:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}

Verify the upstream service is listening on that port with netstat -tulpn | grep 8080. If not, restart the backend service.

# 5. Increase Proxy Buffers and Timeouts

Large responses can overflow default buffers, triggering a 502. Add or adjust these in your Nginx config:

proxy_buffer_size 128k;
proxy_buffers 4 256k;
proxy_busy_buffers_size 256k;

Also increase timeouts to 60-90 seconds to avoid premature closure:

proxy_connect_timeout 60s;
proxy_send_timeout 60s;
proxy_read_timeout 60s;

# 6. Check System Resources

Use top, htop, or free -m to see if memory is depleted. Low memory can kill PHP-FPM workers. If swap is full, consider upgrading RAM or optimizing your application. Also check file descriptor limits with ulimit -n. Nginx and PHP-FPM may hit their limit under heavy load.

# 7. Verify Firewall and Network Rules

If the upstream server is on a different host, ensure Nginx can reach it. Test with curl http://upstream-ip:port. If curl fails, check iptables/ufw rules and cloud security groups. A common oversight is a strict firewall that blocks the Nginx IP.

# Frequently Asked Questions

# What causes a 502 Bad Gateway error in Nginx?

A 502 error appears when Nginx cannot get a valid response from an upstream server. Common causes include a crashed PHP-FPM process, an overloaded application server, misconfigured proxy settings, or network issues between Nginx and the backend.

# How to fix a 502 error in Nginx quickly?

The fastest fix is to restart both Nginx and your backend service: sudo systemctl restart php-fpm && sudo systemctl restart nginx. If that doesn’t work, check the error logs for the specific cause.

# Is a 502 error server-side or client-side?

It is server-side. While the client sees the error, the problem originates from the web server stack (Nginx, backend, or database). The client cannot fix it; the server administrator must.

# How to check Nginx error logs for 502 errors?

Nginx error logs are typically located at /var/log/nginx/error.log. Use tail -f /var/log/nginx/error.log to see errors as they happen. Look for lines containing “upstream” or “502”.

# Can a firewall cause a 502 error?

Yes. If a firewall blocks Nginx’s IP from reaching the upstream server, Nginx will receive no response and display a 502. Check firewall rules on both hosts and ensure required ports are open.

# Should I increase PHP-FPM child processes to fix 502?

If your logs show “server reached pm.max_children setting,” increasing pm.max_children can help, but monitor memory usage first. A sudden spike in traffic may require more workers, but inadequate memory will cause crashes.

# Conclusion

The Nginx 502 Bad Gateway error is a sign that something is wrong upstream. By methodically checking your backend services, logs, configuration, and system resources, you can diagnose and fix it in minutes. Remember to always test configs with nginx -t after changes to avoid syntax errors. With these seven fixes, you’ll be prepared to handle any 502 that comes your way.
